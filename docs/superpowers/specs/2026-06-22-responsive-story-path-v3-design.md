# 「叙廊」响应式故事路径 V3 设计规格

日期：2026-06-22  
状态：已完成交互方案确认，等待书面规格确认

## 1. 背景与目标

当前版本已经支持连续相机进度、故事路径模板、画框和尺寸设置，但仍有四个影响核心体验的问题：

1. 横屏编辑器的 `AppBar` 与章节栏长期占据约三分之一高度。
2. 故事路径由固定 S 曲线和四格随机式模板组合而成，曲线与图片、文字容易重叠，不能表达清晰的先后顺序。
3. 相机控制器只读取横向位移，竖屏编辑预览和播放器仍沿横向移动。
4. 播放器用纵向 `PageView` 切章节、用横向手势浏览章内内容，无法在竖屏中提供符合阅读直觉的纵向连续浏览。

本次改版不改变现有展览、章节、图片和样式数据模型。目标是在保持现有墨色画布、暖白相纸、衬线叙事文字和 2.5D 景深语言的基础上，使故事路径具备明确顺序、方向自适应和连续阅读感。

## 2. 已确认的交互决策

采用方案 A「边界交接」：

- 竖屏时，章内内容沿纵向连续浏览。
- 横屏时，章内内容沿横向连续浏览。
- 到达章尾后，当前手势只停在边界并显示下一章提示；用户结束手势，再开始一次同方向手势才切换下一章。
- 到达章首后采用相同规则返回上一章。
- 横屏中与章内横向轴正交的纵向手势仍可直接切换章节。

边界交接避免了竖屏章内浏览与纵向 `PageView` 争抢同一手势，也能防止一次快速滑动同时越过照片和章节。

## 3. 响应式编辑器布局

### 3.1 竖屏

竖屏保留现有结构：顶部导航栏、62dp 章节栏、画布预览和底部属性面板。顶部导航与章节管理继续保持可发现性，不改变用户已经建立的编辑习惯。

### 3.2 横屏

横屏不再构建常驻 `AppBar` 和常驻章节栏。`Scaffold` 内容区直接铺满安全区域，并使用 `Stack` 放置以下悬浮控件：

- 左上：44dp 返回按钮。
- 右上：44dp 图标按钮组，包含章节、撤销、重做和播放。
- 展览重命名、添加章节和章节排序放入“章节”浮层。
- 章节浮层高度不超过 64dp，覆盖在画布上方而不挤压画布；选择章节或点击画布后收起。
- 右侧属性面板保持独立纵向滚动，宽度继续限制在 280–340dp。

横屏默认状态不允许任何顶部控件参与正文布局，因此画布高度只受系统安全区限制。所有悬浮按钮使用半透明墨色背景和至少 48dp 的实际触控范围，视觉尺寸保持 44dp。

## 4. 有序故事路径布局

### 4.1 主轴规则

故事路径模板根据画布宽高选择唯一主轴：

- `height >= width`：`Axis.vertical`。
- `width > height`：`Axis.horizontal`。

图片的主轴坐标必须按 `Placement` 顺序严格递增。竖屏只允许 X 坐标在受控走廊内左右交替，横屏只允许 Y 坐标在受控走廊内上下交替。尺寸、轻微旋转和深度仍由图片设置与确定性节奏决定，不使用随机数。

### 4.2 间距与层叠

每张图片先根据 `GallerySize` 计算实际尺寸，再计算下一张图片的主轴起点。相邻焦点矩形之间至少保留 24dp 主轴间距。相机过渡中的透视缩放允许最多约 12% 的短暂视觉层叠，但当前焦点图片不得被后一张图片遮住主体区域。

当前焦点图片始终获得最高渲染深度；其他图片按与焦点的距离降低深度和透明度。

次轴位置使用固定交替序列，并限制在安全走廊内：

- 竖屏：图片中心 X 位于画布宽度的 32%–68%。
- 横屏：图片中心 Y 位于画布高度的 34%–66%。

序列超过四张时继续交替推进主轴，不重复使用同一组绝对矩形，因此任意数量图片都保持明确前后关系。

### 4.3 路径几何

`ResolvedNarrativeFrame` 增加当前方向和有序路径锚点。每个锚点来自对应图片焦点矩形边缘附近，而不是使用固定画布百分比。相邻锚点由三次贝塞尔曲线连接，两个控制点的主轴坐标都限制在起点与终点之间，从几何上保证曲线不回头。

路径绘制规则：

- 只连接按 Placement 顺序相邻的节点。
- 线条绘制在所有图片下层，照片自然遮挡路径。
- 仅绘制当前可见窗口及其前后一个节点，避免远距离连线穿过整个画布。
- 节点圆点位于图片外缘，编号与短注释放在锚点的空白侧。
- 标签位置经过画布边界约束；如果与当前图片相交，则切换到另一侧。
- 纸张主题使用低透明墨色线，墨色主题使用低透明暖白线。

由于锚点主轴严格递增，且每段控制点不越过相邻主轴区间，各段不会回折或相互交叉。

## 5. 连续相机与方向自适应

### 5.1 相机模型

新增方向概念 `NarrativeAxis`，由当前视口决定。`NarrativeCameraController` 不再固定读取 `delta.dx`：

- 竖屏读取 `delta.dy / viewport.height`。
- 横屏读取 `delta.dx / viewport.width`。

手指向上拖动表示竖屏阅读进度前进，手指向左拖动表示横屏阅读进度前进；反向拖动则回退。

所有节点共享同一个相机主轴位移，再根据距焦点的距离计算缩放、透明度、旋转和景深。这保证相机移动过程中节点顺序仍然单调，不会出现图片互相穿越。

旋转设备时保留归一化 `cameraProgress`、章节索引和缩放焦点；新的方向只重新解析布局，不重置阅读位置。旋转发生在手势中时结束当前手势锁，下一次触摸按新方向重新判断。

### 5.2 编辑器预览

画布本身成为主要拖动区域：竖屏上下拖动，横屏左右拖动。进度控件只承担定位和状态反馈：

- 竖屏在画布右侧显示窄型纵向进度轨。
- 横屏在画布底部显示横向进度轨。
- 进度百分比与现有相机进度保持一致。

导入按钮与进度轨保持至少 12dp 间距。属性面板的裁切滑块继续只处理自身手势，不传递给画布。

## 6. 播放器手势状态机

播放器移除负责整页章节切换的纵向 `PageView`，改为单一手势协调器。协调器依次处理：缩放、方向锁、章内相机、章节边界。

### 6.1 手势优先级

1. 双指缩放最高优先级。
2. 缩放大于 1.01× 时，单指只平移图片，不修改相机或章节。
3. 1× 状态下，手势开始后依据累计位移锁定主方向。
4. 主轴手势驱动章内相机；正交轴手势只在横屏中切章节。

竖屏中的横向手势不修改章内进度或章节，只保留给系统返回手势和放大后的图片平移。

### 6.2 竖屏边界交接

- 章内相机进度限制为 0–1。
- 到达 1 时显示“继续上滑进入下一章”；到达 0 时显示上一章提示。
- 到达边界的同一次手势不会切章。
- 用户抬手后重新开始手势，并在边界方向累计超过 56dp，才触发章节切换。
- 触发后使用 240ms 淡入与轻微推进动画；开启减少动态效果时改为 120ms 交叉淡化。
- 第一章章首和最后一章章尾不触发切换，只显示终点反馈。

### 6.3 横屏章节切换

横向手势驱动章内相机。纵向累计位移超过 56dp 后切换上一章或下一章，且一次手势最多切换一章。横向和纵向一旦锁定，在本次手势结束前不能互换。

### 6.4 显式控件

上一章、下一章和回到全景按钮继续保留，单击画布仍控制控件显隐。显式按钮调用与手势相同的章节协调器，避免两套状态不同步。

## 7. 组件边界

本次修改集中在以下独立单元：

- `LayoutResolver`：负责方向、尺寸和单调焦点矩形，不处理动画或手势。
- `NarrativeTrackResolver`：把静态布局解析成共享相机轨道和路径锚点。
- `ResolvedNarrativeFrame`：提供当前节点、方向和路径几何，是编辑器与播放器唯一渲染输入。
- `NarrativeCameraController`：按指定主轴更新归一化进度和惯性。
- `NarrativeNavigationCoordinator`（新增）：管理边界预备状态、章节切换阈值和一次手势只切一章规则。
- `SceneCanvas`：渲染照片、数据驱动路径和标签，不自行决定布局。
- `EditorScreen`：只决定响应式界面结构和浮层显隐。
- `ViewerScreen`：把缩放和原始手势事件交给协调器，不再依赖嵌套 `PageView`。

这些单元通过值对象和回调通信，不向布局层传入 Riverpod 或数据库依赖。

## 8. 状态与数据流

1. `EditorSession` 提供当前展览、章节、Placement 和媒体数据。
2. `LayoutResolver` 根据章节与视口生成有序焦点布局。
3. `NarrativeTrackResolver` 生成轴向轨道与路径锚点。
4. 编辑器或播放器手势更新 `NarrativeCameraController.progress`。
5. `SceneCanvas` 使用 progress 解析 `ResolvedNarrativeFrame` 并一次性绘制路径、标签和图片。
6. 到达边界时，播放器把后续新手势交给 `NarrativeNavigationCoordinator`；协调器切章并恢复目标章的已保存进度，首次进入则使用 0。

章内进度只保存在当前播放器会话内，不写入数据库。编辑器每个章节也在当前编辑会话中保留独立预览进度，切换章节后再返回不会跳回 0。

## 9. 异常与边界处理

- 空章节：显示现有空状态，不显示路径和进度轨；主轴边界交接仍按文档顺序进入相邻章节，不自动跳过空章节。
- 单图章节：图片居中，进度固定为 0，主轴手势直接采用边界交接规则。
- 缺失或损坏图片：继续使用现有可恢复占位，仍保留节点和路径顺序。
- 视口宽高为 0：返回空的稳定布局，不执行除法或惯性模拟。
- 手势中旋转屏幕、弹出对话框或离开页面：立即结束方向锁与惯性动画。
- 缩放从大于 1× 回到 1×：必须开始新手势后才能恢复导航，防止缩放结束瞬间误切内容。
- 章节切换动画进行中：忽略新的章节切换请求，但允许动画结束后下一次手势继续导航。

## 10. 测试与验收

### 10.1 单元测试

- 1–8 个节点在竖屏中的中心 Y 严格递增，在横屏中的中心 X 严格递增。
- 相邻焦点矩形满足 24dp 主轴间距，尺寸为小、中、大时仍成立。
- 路径锚点顺序与 Placement 顺序一致；控制点不越过相邻主轴区间；路径无自交。
- 相机控制器在竖屏只响应纵向位移，在横屏只响应横向位移。
- 惯性速度按当前主轴尺寸换算。
- 边界同一次手势不切章；结束后新手势超过 56dp 才切章；一次手势最多切一章。
- 缩放大于 1.01× 时相机和章节不变化。

### 10.2 Widget 测试

- 横屏编辑器不存在常驻 `AppBar` 和章节栏，画布占满安全区；章节浮层可打开、选择和自动收起。
- 竖屏编辑器恢复顶部导航与常驻章节栏。
- 竖屏拖动画布改变相机进度，横屏拖动画布沿横向改变进度。
- 播放器竖屏完成章内浏览和二次手势边界切章。
- 播放器横屏横向浏览、纵向切章。
- 横竖屏切换后章节索引和归一化进度保持不变。

### 10.3 Golden 与人工验收

- 更新故事路径竖屏和横屏的 0%、35%、70% Golden。
- 参考图与新实现使用等比例视口并排比较，检查图片顺序、路径交叉、文字重叠、边界留白和景深层次。
- Android API 29 与 API 36 人工验证横竖屏编辑、播放、缩放、边界切章及系统安全区。
- 横屏默认画布可用高度不得因应用内导航栏减少；悬浮控件不能遮挡当前焦点图片主体。
- 中端 1080p 设备连续拖动目标为 60fps，路径 painter 不在每帧执行图片解码或数据库访问。

## 11. 明确边界

- 本次只重构故事路径模板和相关响应式导航，不改变其他四种模板的版式语义。
- 不增加自由拖拽图片、手动画路径、路径节点编辑或真 3D 相机。
- 不新增数据库字段，不迁移现有展览数据。
- 不改变 v1 的离线、本地图片和无网络权限约束。
- 不在本轮增加云同步、视频、音乐、导出或分享功能。

## 12. Git 提交策略

已将当前可运行版本作为 Git 基线。实施阶段按以下独立提交推进：

1. 轨道、路径几何和相机方向测试及实现。
2. 播放器导航协调器测试及实现。
3. 编辑器响应式布局与预览交互。
4. Golden、设备验证和视觉修正。

每个提交必须在对应测试通过后创建，避免把布局、手势和视觉修正混成不可回退的大提交。
